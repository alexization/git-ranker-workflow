평가 메모:

- 현재 프런트엔드는 “로컬 개발은 가능하지만 harness 반복 실행은 아직 불안정한 상태”에 가깝다.
- build가 네트워크 의존 폰트 fetch에 막히는 점은 `GRC-03`의 핵심 blocker로 봐야 한다.
- 계약 drift와 테스트 부재 때문에 `ranking read` 흐름을 자동 검증하기 위한 선행 작업이 필요하다.

## Cross-Repo Reading

- 첫 하네스 대상인 `랭킹 조회` 기준으로 보면 backend가 가장 앞서 있고, workflow와 client는 아직 supporting asset을 채워야 한다.
- 가장 가까운 blocker는 `client build 재현성`, `backend API 계약`, `backend integration test preflight`다.
- workflow 저장소는 실행 런타임과 evidence loop가 아직 없어서, 후속 하네스 작업 전에 문서와 runtime control-plane을 더 채워야 한다.

1. `GRB-01`: `/api/v1/ranking` 중심 OpenAPI 생성 기반을 먼저 고정한다.
2. `GRB-02`: Docker preflight와 integration test 오류 메시지를 정리해 환경 문제와 코드 문제를 분리한다.
3. `GRC-01`: 프런트 계약 타입과 validation schema를 단일화해 drift를 제거한다.

5. `GRW-03`, `GRW-04`: workflow 문서에 backend domain/ops와 frontend data flow를 수집해 Agent 탐색 비용을 낮춘다.
